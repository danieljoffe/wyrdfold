# wyrdfold-api holistic audit ‚Äî 2026-06-22

Multi-agent audit of `apps/wyrdfold-api` + `supabase/migrations` at commit `be98601`
(head of PR #28, `#5 P3`). 12 dimension-scoped finders ‚Üí each finding faced a 3-lens
adversarial panel (reachability / exactness-or-already-mitigated / severity); only
findings that passed a majority are reported. Everything was read-only.

## How this was produced (two passes)

- **Pass 1** (full fan-out, 197 agents / ~5.1M tokens) covered **security + DB** and hit a session limit before the perf/async dimension could be verified.
- **Pass 2** (cost-optimized tail: dedup-first, per-file batched Haiku verifiers, info skipped ‚Äî **28 agents / ~1.08M tokens**) recovered the **performance / async-correctness / FastAPI** dimension. See the [Performance / async correctness](#performance--async-correctness-second-pass) section.

Every finding below passed adversarial verification (security/DB: 3-lens panel; perf: per-file batch + a 2nd independent check on every high). The 24 security/DB findings were **not** re-verified in pass 2.

Two themes dominate:
1. **Security/DB:** the new shared collaborative target/voting/learning model + the right-to-erasure feature both leak across the tenant boundary ‚Äî retaining/exposing per-user data, or letting one user mutate a resource every other user reads.
2. **Performance:** the connection-pooling / parallelism gains from the June 6-12 pass are **undercut by code that shipped since** ‚Äî ~25 handlers block the event loop, and the LLM/embeddings clients are rebuilt per request (defeating the reuse `wyrdfold_bench.py` measured).

## HIGH

### H1 ‚Äî Account deletion does not erase `reference_jds` (user-pasted JD text + `user_id` survive)
- **File:** `app/services/account_deletion.py:53-69` (`_USER_ID_TABLES`)
- `reference_jds` has `user_id` (added in `migrations/20260622120000`) and a NOT NULL `jd_text` of free-form, user-pasted JD text. Written attributed to the caller at `routers/targets.py:1225` (`crud.add_reference_jd`). Deletion's `_USER_ID_TABLES` omits it and there's no scrub ‚Äî so after `DELETE /profile/account`, the user's pasted text + id persist indefinitely. The code's own `crud.delete_reference_jd:681-699` (scoped by `user_id`) proves these are owned/personal rows.
- **Impact:** Right-to-erasure (#29) is incomplete; retained `jd_text` + `user_id` is directly attributable to the deleted person.
- **Fix:** Add `reference_jds` to the erasure cascade as a `user_id`-scoped delete (and re-merge affected targets' profiles), **or** scrub `jd_text`/null `user_id` if the de-biased contribution must remain. Add it to `data_export._EXPORT_TABLES` to keep lockstep.

### H2 ‚Äî Shared scoring profile is cross-tenant data-poisonable via an attacker-supplied reference JD
- **File:** `app/routers/targets.py:1163-1250` (`add_reference_jd`); merge `services/targets/merge.py:65-78`
- Targets are explicitly **shared** (junction `user_targets`); `scores` rows are keyed `(job_posting_id, target_id)` with **no `user_id`** (`target_scoring.py:120`). Any authenticated user can enumerate every target (`GET /targets`) and link to any of them (`POST /targets/{id}/link` ‚Äî existence check only, `targets.py:767-801`). Once linked they pass `_require_user_owns_target` and `POST /targets/{id}/reference-jds` with arbitrary `jd_text` (50-char floor / 100k cap). The LLM-derived profile is union-merged into the **shared** `scoring_profile` (`merge_reference_jds`, keeps most-negative weight) and bumps `profile_version`, re-scoring the shared rows every co-pursuer reads.
- **Impact:** One user can demote/hide/boost postings for every other user pursuing the same target. By-contributor de-bias only equalizes *count*; it doesn't stop one contributor injecting content. P3 vote suppression needs a net quorum of 3 downvotes, so poison lands immediately and persists.
- **Fix:** Treat contribution to a shared target as privileged (require actual creator/owner, not merely linked), or quarantine a new JD's profile influence behind a positive vote gate **before** merge. At minimum cap per-contribution keyword counts + weight magnitudes pre-merge; reconsider per-`(user,target)` scores.

### H3 ‚Äî Resume upload parses attacker DOCX/PDF with no decompression-bomb guard ‚Üí single-request OOM
- **File:** `app/services/ingest/parse.py:66-85` (`parse_docx`), `:37-63` (`parse_pdf`); reached from `routers/experience.py:206-213`
- `POST /experience/upload-resume` caps only the **compressed** body at 10 MB, then `Document(io.BytesIO(file_bytes))` inflates the full OOXML zip into an lxml tree ‚Äî a deflate-bomb `document.xml` expands to multiple GB and OOM-kills the process. No inflated-size / member-count / ratio check. `parse_pdf` iterates `pdf.pages` with no page cap (analogous CPU/RAM blowup).
- **Impact:** Any authenticated user (open magic-link signup post-launch) can OOM/CPU-pin the single Railway replica with **one** ~10 MB upload. The 3/min rate limit doesn't help ‚Äî one request suffices.
- **Fix:** Before parsing, open the zip with `zipfile`, sum `ZipInfo.file_size`, abort on inflated total / per-member cap (e.g. 50-100 MB) or ratio > ~100√ó; cap member count. For PDF, reject when `len(pdf.pages)` exceeds a cap (~50) before extracting.

## MEDIUM

### M1 ‚Äî `DELETE /jobs/{posting_id}` hard-deletes the globally-shared posting after only a follower check
- **File:** `app/routers/jobs.py:1749-1766`
- `_assert_user_owns_posting` only checks the caller follows *some* target that scored the posting, then runs an unscoped `jobs.delete().eq('id', posting_id)` on the service-role client. Any one follower can delete a shared posting (and its scores) for all users tracking it.
- **Fix:** Restrict hard-delete to operator/api-key callers, or make user "delete" a per-user hide via a `user_jobs` status.

### M2 ‚Äî DNS-rebinding window between `assert_safe_host` resolution and httpx connect on user-supplied URLs
- **File:** `app/http_client.py:159-176`; `services/validate.py:112-136`; reached via `/jobs/manual`, `/targets/from-url`, `/targets/{id}/reference-jds`
- The guard resolves the hostname, checks no resolved IP is internal, then hands the **hostname** to httpx, which re-resolves at connect. A TTL-0 domain can answer the validation lookup with a public IP and the connect lookup with `169.254.169.254`/`127.0.0.1`/RFC1918. Code acknowledges the gap (`validate.py:121-124`) but doesn't mitigate. Capped at Medium: needs an authoritative TTL-0 server + a sub-second race.
- **Fix:** Pin the validated IP at connect time (resolve once, assert safe, connect to that exact IP via custom transport/resolver with original Host/SNI).

### M3 ‚Äî Account deletion + export both omit `contribution_votes`
- **File:** `app/services/account_deletion.py:53-69`
- `contribution_votes` (`migrations/20260622140000`) has `user_id` NOT NULL recording each user's up/down votes; written by `services/targets/votes.py:23-44`. It's in neither `_USER_ID_TABLES` nor `_EXPORT_TABLES`, and its `ON DELETE CASCADE` is off `reference_jds` which is itself never deleted (see H1). Vote history (with `user_id`) survives erasure and is missing from export.
- **Fix:** Add `contribution_votes` to both inventories (`user_id` is TEXT, so `.eq('user_id', user_id)` works directly).

### M4 ‚Äî Unescaped JD text + example-title pools flow into LLM prompts ‚Üí prompt injection of the shared title-triage gate
- **File:** `app/services/relevance/title_triage.py:167-208`; `derive_profile.py:283`; writeback `targets.py:1242-1246`
- `derive_profile_from_jd` passes raw `jd_text` as the entire user turn (no delimiting). The derived `example_promising/unpromising_titles` have no per-element length or count cap (`models/targets.py:394-395`) and are written verbatim onto the **shared** target from the latest JD. `title_triage` interpolates those attacker-influenced examples **above** the untrusted candidate titles in the Phase-1 gate that decides which jobs every linked user ever sees ("a false-negative here is lost forever").
- **Impact:** Cross-tenant prompt injection of the ingestion relevance gate ‚Äî suppress/force-admit postings for all co-pursuers. Forced `complete_json` constrains output shape, not which verdicts the model is steered to emit.
- **Fix:** Wrap `jd_text` in an instruction-inert block; enforce length/count caps + strip newlines on `example_*_titles` before storing; delimit the example block from the candidate block in `title_triage`.

### M5 ‚Äî Polled job titles (untrusted external ATS content) feed the shared feedback learner that mutates the shared negative list
- **File:** `app/services/llm_learner.py:91-117, 137-151, 213-256, 451-464`
- The learner builds its prompt from job titles fetched from the `jobs` table (originating from polled external ATS pages ‚Äî attacker-influenceable) + user feedback reasons, serialized raw with no delimiting. High-confidence `ProfilePatch` auto-applies `add_negative` to the **shared** `scoring_profile` and writes back to the shared target. The #5 P4 rate-cap bounds churn *magnitude* but not the *direction/content* of an injected negative keyword.
- **Fix:** Delimit/label untrusted title + reason text; validate/cap learner-proposed `add_negative` keywords against an allowlisted shape pre-apply; consider scoping learner output to the contributing user.

### M6 ‚Äî Resume parsing runs in a threadpool worker with no timeout ‚Äî a pathological file hangs a worker forever
- **File:** `app/routers/experience.py:213`
- `await asyncio.to_thread(parse_resume, ...)` dispatches CPU-bound attacker-input parsing with no `wait_for`/cancellation/CPU bound. Threadpool threads can't be force-cancelled, so a hung parse permanently consumes a worker ‚Äî and the codebase routes most sync supabase `.execute()` through the same pool, so a few slow uploads starve all DB work. Compounds H3.
- **Fix:** Parse in a `ProcessPoolExecutor` with a hard `asyncio.wait_for` wall-clock timeout; bound concurrent parses with a semaphore.

## LOW

- **L1 ‚Äî Shared learnable profile mutable by any authenticated follower (profile poisoning).** `routers/targets.py:1163-1250`. Largely by design (collaborative profiles) with de-bias + downvote-quorum mitigations; residual risk amplified if cheap/anonymous identities exist. Gate contribution + vote weight to non-anonymous accounts; rate-limit per-`(user,target)` and cap profile_version movement.
- **L2 ‚Äî Shared httpx client follows redirects with no host validation.** `http_client.py:35` (`follow_redirects=True`), used by ATS probers / `GET /sources/verify` / `url_health` / notify. First hop is host-constrained (no direct anon SSRF), but an upstream 3xx into an internal range would be followed without the guard. Set `follow_redirects=False` and follow manually with `assert_safe_host` per hop (as `get_with_size_cap` already does).
- **L3 ‚Äî Deletion/export lockstep test compares the two lists only to each other, never to the schema.** `tests/test_data_export.py:138-140`. This is the **root-cause control gap** behind H1 + M3 ‚Äî both lists were edited together and both omit the same two tables, so the assertion stays green. Add a test that parses migrations (or queries `information_schema`) for every table with a `user_id` column and asserts each is deleted, cascade-covered, or consciously allow-listed; fail CI otherwise.
- **L4 ‚Äî `GET /targets` returns every user's targets (cross-user metadata exposure + enumeration primitive for H2).** `routers/targets.py:360-365` ‚Üí `crud.list_all_summary`. The sibling `get_active` was deliberately scoped to avoid this exact leak; the bare list was left global. Scope `list_targets` to the caller for JWT callers; keep instance-wide only for api-key/operator.
- **L5 ‚Äî `jd_url` on `/targets/from-url` + `/targets/{id}/reference-jds` has no `max_length`** (unlike the 2048-capped `/jobs` URL inputs). `models/targets.py:320,333`. Not an SSRF/OOM bypass (guards still apply), just an unbounded-string inconsistency. Add `Field(max_length=2048)`.
- **L6 ‚Äî `reference_jds` exposes every user's JD text + contributor `user_id` to all authenticated callers.** RLS SELECT policy `USING (true)` TO authenticated (`migrations/20260614120000:25-26`). The browser-shipped anon key makes `authenticated` reachable directly at PostgREST, so a direct read re-identifies who contributed which JD ‚Äî undercutting the "anonymous votes" guarantee. Replace with a follower-scoped policy, or drop `user_id` from what the authenticated SELECT returns.
- **L7 ‚Äî Shared-catalog SECURITY DEFINER writers have zero in-body authorization.** `remote_schema.sql` (`bulk_update_*`, `insert_source_if_not_exists`), `bulk_update_url_health.sql`, `poller_scoring_rpcs.sql` (`archive_jobs_by_ids`). Safety rests **entirely** on grant revocation (anon/authenticated currently revoked ‚Üí not reachable today). `search_path` pinned, no dynamic SQL. But `archive_jobs_by_ids` shipped with `GRANT ALL TO anon` and was closed only by a later migration, and Postgres re-adds PUBLIC EXECUTE on every `CREATE` ‚Äî one grant slip turns these into an anon shared-catalog corruption primitive. Add an in-body `IF auth.uid() IS NOT NULL THEN RAISE insufficient_privilege` guard (mirroring `user_apply_score_blend`); add a CI assertion over `pg_proc` ACLs.

**Latent / hygiene:**
- **I1 ‚Äî BYOK ciphertext not bound to `user_id` via GCM AAD.** `services/keys/crypto.py:69-95`, `associated_data=None`. Not reachable (service-role-only table, always selected by `user_id`); matters only under a separate DB-write compromise. Pass `associated_data=f"{user_id}:{provider}"`.
- **I2 ‚Äî `wyrdfold_beta_invites` retains user email after deletion.** Invite ledger keyed by email, no `user_id`, appears unused at runtime. Decide: delete on erasure or document as intentionally-retained.
- **I3 ‚Äî `data_export` docstring wrongly says `experience_chunks` "carries no user-authored content"** ‚Äî it stores resume-derived text. Not a defect (cascade-deleted + content exported elsewhere); fix the misleading comment.
- **I4 ‚Äî Hardening migrations miss 3 trigger/event-trigger DEFINER functions** (`rls_auto_enable`, two `set_*_updated_at`) still holding `GRANT ALL TO anon/authenticated`. Exploitability-inert (Postgres blocks direct calls to trigger-returning functions) but breaks the "every DEFINER function locked down" invariant. `REVOKE ALL ... FROM PUBLIC, anon, authenticated` for consistency.

**Verified-safe (positives, recorded so the real findings aren't diluted):**
- **I5 ‚Äî SSRF defense, pandoc shell-out, CORS, JWT auth all verified sound.** User-URL fetchers reject loopback/link-local/RFC1918/metadata + bare IPs, follow redirects manually with per-hop re-check, stream with a 5 MB cap. `pandoc` uses fixed argv + stdin (no injection; style inputs are Literal enums). CORS is an explicit allowlist with `allow_credentials=False`. JWT uses JWKS with required `exp`/`sub`/`aud`/`iss`; prod error bodies generic.
- **I6 ‚Äî Overall RLS/grant posture is strong; no critical/high RLS holes on the focus tables.** `contribution_votes`, `user_jobs`, `user_targets` are `FOR ALL` with `USING`+`WITH CHECK`; `user_api_keys` is service-role-only with anon/authenticated revoked; the public anon key is auth-only (no `.from()`/`.rpc()` in the frontend). Adopt the `contribution_votes` policy shape as the standard.
- **I7 ‚Äî User-facing DEFINER write RPCs are correctly built** (`user_apply_score_blend`, `user_upsert_score`, `user_set_scores_included`): pinned `search_path`, `(select auth.uid())::text = user_id` ownership gate, service-role exemption, PUBLIC revoked, no dynamic SQL. Use as the template for retrofitting L7.
- **I8 ‚Äî Reachability assumption confirmed:** the anon key is browser-distributed, so `anon`/`authenticated` roles ARE reachable by untrusted clients at PostgREST. This is *why* the DEFINER grant lockdowns are load-bearing and a re-grant would be a real High.

## Performance / async correctness (second pass)

17 findings confirmed (**13 high, 4 medium**). The 13 highs are mostly one systemic pattern restated across routers ‚Üí ~6 distinct issues below. This is the dimension tied to the June 6-12 optimization pass, and the headline is blunt: **the pooling/parallelism wins are being undercut by event-loop blocking and per-request client rebuilds in code that shipped since.**

### P-H1 ‚Äî Systemic: ~25 `async def` route handlers call synchronous supabase-py directly on the event loop
- **Files:** `routers/feedback.py` (create/remove/list_feedback, run_learner_now, apply/reject_learning), `routers/tailor.py` (~10 handlers incl. `create_tailored_resume`), `routers/targets.py` (activate_target, the `_activate_pipeline` BackgroundTask, add/delete/vote reference-jd, axis-weights, notification-thresholds), `routers/experience.py` (prose/preferences/turns), `routers/jobs.py` (`rescore_for_target`).
- supabase-py is **synchronous**; an unwrapped `.execute()` inside an `async def` runs on the single event-loop thread and **freezes every concurrent request across all tenants** for that DB round-trip (~20-100ms, worse under load). The repo *has* the right convention (the `#107` "use plain `def` so FastAPI threadpools it" comment, followed correctly in `user_profile.py`, `analysis.py`, `jobs.py` reads) ‚Äî ~25 handlers don't. Worst: `feedback.run_learner_now` runs the **entire multi-query learner inline** on the loop; `create_tailored_resume` (hottest path, 30/min) wraps *one* call in `to_thread` but leaves 3 more blocking ‚Äî a false sense of safety.
- **Fix:** Pick one rule repo-wide ‚Äî all DB-touching handlers plain `def` (threadpool), or every supabase call via `await asyncio.to_thread(...)`. Add a CI grep/AST check flagging `async def` route handlers with a bare `.execute()`. (Root cause; the per-router items are instances.)

### P-H2 ‚Äî Bulk resume export: N blocking Supabase Storage downloads serialized on the loop
- `routers/tailor.py:390-432`. `export_resumes_zip` loops `persistence.get` per id (N+1 DB), then `download_docx` per resume (blocking Storage HTTP), all inline. A 20-resume export = 20 serial blocking downloads that freeze the worker. **Fix:** `asyncio.gather` the per-record `to_thread(download_docx, ...)`.

### P-H3 ‚Äî LLM & embeddings clients rebuilt per request (defeats the pooling you benchmarked)
- `services/llm/__init__.py:45,69,100` via `dependencies.get_llm_client`/`get_embeddings_client`. Every analysis/targets/tailor/feedback/experience request constructs a **fresh** `AsyncAnthropic`/OpenRouter/Voyage client ‚Äî a new TCP+TLS pool to the upstream ‚Äî and never closes it. The exact opposite of the +49% connection-reuse win `wyrdfold_bench.py` measured; leaks sockets/FDs until GC. **Fix:** build default client(s) once at lifespan startup and reuse (BYOK per-user clients cached by user-id + `aclose`d); at minimum `lru_cache` `get_default_client()` and close on shutdown.

### P-H4 ‚Äî `suggest_and_match`: N+1 of two blocking DB round-trips per LLM suggestion
- `services/targets/match.py:90-124`. After 5-10 LLM suggestions, loops `find_matching_target` per suggestion ‚Üí 2 blocking `.execute()` each ‚Üí 10-20 serial round-trips on the loop in a user-facing request. **Fix:** bulk-resolve via one `.in_('normalized_label', labels)` + one batched fuzzy RPC, match in memory.

### P-H5 ‚Äî `rescore_for_target`: DB-heavy + CPU-bound bulk re-score on the event loop
- `routers/jobs.py:1549,1565`. `bulk_score_for_target` (O(jobs √ó keywords) pure-Python + many `.execute()`) runs on the loop. Operator-gated (`verify_api_key`) caps reach, but the CPU loop must not run on the event loop. **Fix:** plain `def` (no awaits) or `to_thread`.

### Medium
- **P-M1 ‚Äî Re-score runs as a non-durable `BackgroundTask`** (`feedback.py:106-108,270-309`). User feedback expects a re-rank; on deploy/crash between response and task it's silently dropped. Persist intent (mark `profile_version` stale / enqueue a row) so the scheduler reconciles it.
- **P-M2 ‚Äî Poller N+1** (`poller.py:715-728`): fetches each user's optimized doc one-by-one in a nested loop per poll. Batch via `.in_('user_id', ids)` or `gather`.
- **P-M3 ‚Äî `list_user_targets_with_summary` over-fetch** (`crud.py:314-348`): `select("*")` pulls full `scoring_profile` JSON + title pools just to project to a light summary. Use an explicit column list.

> Method note: dedup-before-verify didn't merge the 3 independent `feedback.py` reports (different finder titles ‚Üí different keys), so they appear as corroboration ‚Äî but per-file batched verification still checked them in one agent, so the cost saving held.

## Suggested priority order

1. **P-H1 + P-H3 (throughput-wide)** ‚Äî event-loop blocking across ~25 handlers + per-request client rebuild. Caps the whole API's concurrency on the single replica and directly undercuts the June 6-12 pooling wins. One convention + CI check fixes the class.
2. **H3 + M6** ‚Äî resume-upload DoS (zip-bomb guard + process-isolated parse with timeout). One open endpoint, single replica = whole-API outage.
3. **H1 + M3 + L3** ‚Äî erasure gaps + their root-cause (schema-grounded inventory test). One coordinated fix closes both leaks and prevents recurrence.
4. **H2 + M4 + M5 + L1** ‚Äî shared-target poisoning surface (contribution gating, prompt delimiting, learner input caps). Largest blast radius; needs a design decision on the collaborative model.
5. **P-H2 + P-H4 + P-H5 + P-M2** ‚Äî N+1 / loop-stall hotspots (export, suggest, rescore, poller).
6. **L4 + L6** ‚Äî cross-user exposure via `GET /targets` and `reference_jds` RLS. **L7 + I4** ‚Äî defense-in-depth on DEFINER writers + CI ACL assertion.

## Deeper async/perf sweep (coverage-critic follow-ups, not yet verified)

The perf critic flagged these issue *classes* as worth a dedicated pass ‚Äî surfaced, not confirmed:

- **Request-cancellation / client-disconnect** ‚Äî long-running stream/bulk/LLM handlers don't check `request.is_disconnected()`; abandoned requests waste CPU/tokens.
- **Streaming error boundaries** ‚Äî mid-stream DB/LLM failures in `StreamingResponse` have no `try/finally` cleanup (in-flight connections/LLM calls leak).
- **`asyncio.gather` error handling** ‚Äî inconsistent `return_exceptions` (set in `source_discovery`, absent in `smartrecruiters`/`workday`/`poller`) ‚Üí silent partial failures.
- **Threadpool / backpressure** ‚Äî many `to_thread` DB calls share the default threadpool with no global ceiling; under load, requests √ó DB calls can starve it. Audit threadpool sizing vs `LLM_CONCURRENCY`.
- **HTTP-layer body-size limit** ‚Äî 10MB cap is app logic, not Starlette `max_request_size`; oversized JSON consumes memory before validation.
- **Scheduler shutdown drain** ‚Äî verify cron/background tasks drain gracefully on shutdown (only `cost_log_buffer` uses the cooperative pattern).
- **Concurrent-write races** ‚Äî feedback vote tallying / target activation / contribution votes lack explicit transaction scoping or row locks.
- **Large-export memory** ‚Äî bulk zip / large derived docs buffer the full response in memory rather than streaming chunks.

## Verified-safe (perf)
- `poller.py` handles blocking DB (via `to_thread`), `gather` fault-isolation, and background-task lifecycle **correctly** ‚Äî it's the in-repo model for the fixes above (alongside `user_profile.py` and `analysis.py`).
