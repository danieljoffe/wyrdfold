# RLS data-access map (#88 / #6)

Which Supabase client each API surface runs on, and **why the service-role
paths are service-role on purpose**. This is the finish line for #88: when a
path below says "intentional", migrating it is not pending work тАФ changing it
needs a design decision, not a cleanup PR.

Two clients exist (`app/dependencies.py`):

| Client                                                   | Dependency                                                                                                        | RLS                     |
| -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- | ----------------------- |
| **User client** тАФ per-request, bound to the caller's JWT | `get_user_supabase` (JWT-only) / `get_supabase_for_caller` (dual-auth: JWT тЖТ user client, api-key тЖТ service-role) | enforced (`auth.uid()`) |
| **Service-role** тАФ singleton, bypasses RLS               | `get_supabase`                                                                                                    | bypassed                |

## Intentionally service-role тАФ with the reason

| Surface                                                                                                                                                                    | Why it cannot / must not use the user client                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Cost ledger writes** (`llm_costs` in derive/consolidate/turn/probe/chunks)                                                                                               | `llm_costs` deliberately has **no INSERT policy** for `authenticated`: a user able to write cost rows could forge negative costs and bypass budget enforcement. Endpoints thread a separate service-role `cost_supabase` alongside the RLS client (the Phase-1 dual-client pattern).                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **Shared-catalog writes** (`jobs`, `targets`, `scores`, `reference_jds` тАФ learner endpoints in `feedback.py`, target CRUD/activation in `targets.py`, manual-add, rescore) | The shared catalog has read-only policies and **no write policy by design** тАФ N users share one target/posting, so "the caller's rows" is undefined. Writes go through service-role + Python ownership guards, or SECURITY DEFINER RPCs (`user_apply_score_blend`, vote tally). The open decision on hardening this further is **#6 R2**.                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Operator / cron surfaces** (`admin.py`, `poll.py`, `sources.py`, `discovery.py`, rescore/backfill in `jobs.py`)                                                          | Api-key callers have no JWT тАФ there is no user to bind. Gated by `verify_api_key` instead.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Account deletion** (`user_profile.py DELETE /account`)                                                                                                                   | Spans every per-user table **plus** the Supabase auth admin API (deleting the `auth.users` row) тАФ inherently privileged.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Data export тАФ the three gap reads only** (`user_profile.py GET /export`)                                                                                                 | The export rides the caller's user client (see "Migrated"), with a service-role client threaded alongside (`data_export._SERVICE_ROLE_TABLES`) for exactly three reads that break under RLS: `user_api_keys` and `notifications_sent` have **no `authenticated` grant at all** (a user-client read hard-fails 42501 and would 500 the export тАФ the no-grant posture is deliberate: key material / notification internals are service-role-only), and `reference_jds`' only SELECT policy is **follower-scoped**, so the user's own contributions on unfollowed targets would silently vanish from their export (right-to-access covers what they authored). Pinned by `tests/integration/test_rls_export.py`, incl. dual-vs-service ZIP equivalence. |
| **Pre-auth surfaces** (`waitlist.py`, `keys.py` validation)                                                                                                                | No JWT exists yet at call time.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Background pipeline** (poller, scheduler, batch, learner internals)                                                                                                      | No request context тАФ these run from cron/queue with the service key, charging work to the activating user via explicit `user_id` params.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |

## Dual-auth (JWT тЖТ RLS, api-key тЖТ service-role): deferred, with reason

`jobs.py` list/read surfaces (`GET /jobs`, `/pipeline-counts`, `GET /{id}`) and
`analysis.py` remain on `get_supabase` + `get_current_user_id_optional` even
though a JWT caller _could_ be routed through the user client via
`get_supabase_for_caller`:

- The list path calls the `get_target_jobs` **RPC** тАФ routing it through the
  user client needs an EXECUTE grant + in-function scoping audit first.
- It is the hottest path in the app; flipping its client should be validated
  against a live stack, not shipped blind.

If you pick this up: switch the dependency to `get_supabase_for_caller`,
verify the RPC grant, and extend `tests/integration/test_rls_jobs_reads.py`
to run the list through a user JWT.
